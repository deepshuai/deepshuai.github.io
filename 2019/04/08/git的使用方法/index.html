git的使用方法

发表于 2019-04-08 | 分类于 版本控制

写在最前面

  git之前有使用过，但多局限于一些简单的指令，对其更深更全面的背后逻辑一直没有做深入的理解，这里希望可以通过一篇博客来touch一下git最重要的东西。
  这篇博客主要是学习git的命令以及使用方法，中间会穿插一些版本控制的原理方便理解，但不是本博客的主要目的。计算机是一个逻辑性很强的学科，其逻辑性在版本控制如git中体现十分明显，在学习git使用过程中始终把握最基本的逻辑会起到事半功倍的效果。

  Git不是简单的一篇博客能够总结的清楚的，本博客只是最简化的git使用方法，这里有一个写的非常的清晰和详细的文档Pro Git(中文版)可以参考和进一步的学习。

认识版本控制系统

版本控制系统分为三类：
1、本地版本控制系统： 如rcs，本地保存并管理补丁(patch)，文件补丁是一种特定的文件格式，记录文件修订前后内容的变化。适合个人本地版本管理，使用场景比较有限。
2、集中化的版本控制系统（ Centralized Version Control Systems，简称 CVCS ）： 如CVS，Subversion 以及 Perforce等，集中式顾名思义有一个单一的集中管理的服务器，保存所有文件的修订版本，而协同工作的人们通过客户端连接到这台服务器上，取出最新的文件进行修改并提交更新。好处：更容易控制权限，任务分工更明确；坏处：单点故障问题；单队列的任务提交形式效率很低。
3、分布式版本控制系统（ Distributed Version Control System，简称 DVCS ）： 如Git，Mercurial，Bazaar，Darcs等，不同于集中式版本控制库仅有中央服务器有一份完整的备份，客户端会提取代码仓库的完整备份。

Git文件四种状态

  这是一个很重要的概念，后续的操作主要就是对文件的状态做修改或者在文件状态的基础上执行操作，是理解Git背后逻辑的关键。四种状态分别是：未被追踪的(untracked)，已修改(modified)，已暂存(staged)，已提交(committed)，四种状态的文件流转对应三个工作区域：Git的工作目录，暂存区域，本地仓库。
未被追踪的(untracked)： 通常是在工作目录下自行添加的文件；
已修改(modified)： 对已经在暂存区或者在git 仓库中的文件作出修改，那么文件会被标记为已修改的文件，已修改的文件通常放在工作目录下，等待放入暂存区。
已暂存(staged)： 将未被追踪/已修改的文件通过git add命令将文件添加到暂存区，暂存区的文件通常放在Git目录下，工作目录下已修改的文件通过git add 命令将文件放在暂存区。
已提交(committed)： 暂存区的文件通过git commit命令永久的存储在Git目录中。

Git 基础命令

Git的配置命令

Git仓库的初始化：

git init

配置Git仓库的用户信息：

git config --global user.name xxxx
git config --global user.email xxxx

Git的基础命令

1、将未被跟踪的/已修改的文件提交到暂存区：

git add <file>...

2、将文件提交到本地版本库：

git commit -m "提交描述"   //将暂存区的文件提交到版本库
git commit -am "提交描述"  //将工作区&暂存区所有的文件提交到版本库

3、查看git仓库中的文件状态：

git status

4、查看提交到git仓库中的历史记录：

git log                //提交历史的详细信息,包含校验码和用户信息、提交时间和提交描述等信息
git log --oneline      //每个记录占一行，包含校验和以及提交描述
git log -p -2          //-p展开显示每次提交的内容差异，-2仅显示最近的两次更新记录
git log --since=时间点  //显示指定时间之后的提交记录
git log --until=时间点  //显示指定时间之前的提交记录
git log --author       //仅显示指定作者的提交记录
git log --committer    //仅显示指定提交者的提交记录

5、对比未暂存和已暂存的更新

git diff                        //工作目录当前文件和暂存其区域快照之间的区别
git diff --cached/--staged      //已暂存的文件和版本库之间的差别
git diff 版本号 版本号            //两个版本的差异
git diff 分支 分支               //两个分支的差异

Git的撤销操作

1、修改最后一次的提交：

git commit --amend 
//撤销上一次的提交，并合并暂存区的文件一起再一次提交 
//如果执行命令之前暂存区没有文件，那就相当于仅仅对提交描述做修改而不修改文件内容

2、取消已经暂存的文件：

git reset HEAD <file>... 
//将文件由已暂存的状态转化为已修改的状态
//将文件从版本库拉回到暂存区中

3、取消对文件的修改：

git checkout -- <file>... //撤销对工作区文件的修改

git的删除操作

1、删除工作区和暂存区内的文件

git rm <file>... 
<==>
rm <file>... 
git add <file>...  

git rm --cache <file>...  // 仅仅删除暂存区内的文件而不删除工作区内地文件
git rm -f <file>...       // 工作区和暂存区内的文件同时删除

2、移动文件

git mv <file1> <file2>    // 对暂存区内的文件重新命名
<==>
mv <file1> <file2>
git rm <file1>
git add <file2>

Git分支

git分支的基本命令

1、查看分支

git branch                      //查看分支

2、创建分支

git branch 分支名               //创建分支

3、跳转分支

git checkout 分支名         //切换到相应的分支

4、删除分支

git branch -d 分支名        //不能删除当前所在的分支

5、创建一个分支并切换到该分支

git checkout -b 分支名      //创建并切换分支

6、合并分支文件

git merge 分支名            //有冲突的时候要手动删除，无冲突时可以直接快进制合并

git存储的变更

由于在分支切换的时候，必须要暂存/提交之前分支的修改，除了传统的方式
1、暂存分支工作区的修改

git stash   //将改变封存并删除工作区的修改

2、当前分支下被暂时封存的改变的列表

git stash list

3、在当前的分支下拉回被暂时封存的修改

git stash apply stash@并没有被删除
git stash pop stash@

Git的远程仓库

这里在github上配置远程仓库的过程参考博客：git安装配置以及本地版本库内容上传到github

当多个开发者共同开发某个项目的时候，这时候就可以进行分布式的开发了，主要执行的就是拉取(git pull)和推送(git push)操作，如下：

1、甲开发者将本地的git仓库中的文件推送到远程仓库

git push 远程仓库的HTTPS/SSH

2、乙开发者如果想在项目的基础上做开发，就需要克隆远程仓库

git clone 远程仓库的HTTPS/SSH

3、乙开发者修改之后，将修改推送到远程仓库

git push 远程仓库的HTTPS/SSH

4、甲开发者如果要继续对项目进行开发，需要同步远程仓库

git pull 远程仓库的HTTPS/SSH

5、甲开发者修改之后，将修改推送到远程仓库

git push 远程仓库的HTTPS/SSH
